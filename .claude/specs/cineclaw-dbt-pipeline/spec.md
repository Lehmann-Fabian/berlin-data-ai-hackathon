## Acceptance Criteria

1. (AC-1) `int_fct_clickouts` exists, is event-grain, includes keys required for joining to titles/providers, and filters bots (`not is_bot`) while restricting to clickouts (`se_category='clickout'`).
2. (AC-2) Title and provider dimensions exist in intermediate and marts (`int_dim_titles`, `int_dim_providers`, `dim_titles`, `dim_providers`), and `int_bridge_title_genres` supports genre filtering.
3. (AC-3) `mart_popular_movies_dataset_period` exists as a dbt-refreshed table and includes dataset min/max timestamps plus popularity measures (clickouts, unique_users, unique_sessions) at grain (app_locale, provider, movie).
4. (AC-4) A semantic model definition exists over `fct_clickouts` and includes measures `clickouts`, `unique_users`, `unique_sessions` and core dimensions for Lightdash/ad-hoc analysis.
5. (AC-5) dbt tests exist: PK uniqueness/not_null for clickouts (`rid`), relationships for title/provider keys, and accepted-values checks for `platform`, `event`, and `se_category` on `fct_clickouts`.
6. (AC-6) `mart_movie_popularity_rolling_29d_top20` exists as a dbt-refreshed table and provides Top 20 movies per (anchor_date, app_locale, genre_bucket) based on clickouts in the ¬±14 day window.
7. (AC-7) `mart_movie_popularity_seasonal_ddmm_top20` exists as a dbt-refreshed table and provides Top 20 movies per (anchor_ddmm, app_locale, genre_bucket), aggregating weighted ¬±14 day windows across all dataset years.

## Non-Functional Requirements

- Performance: Default intermediate models must be cheap to iterate on (default to T1-derived staging via `vars.events_model` override) and allow scaling by switching the referenced staging model.
- Security/Privacy: No PII enrichment. Use only existing anonymized identifiers (`user_id`, `login_id`) and keep transformations within Snowflake via dbt.
- Reliability/Observability: Prefer deterministic transforms; enforce key constraints and relationships via dbt tests.

## Edge Cases

- `app_locale` can be NULL for some events; popularity mart should not assume it is always present.
- Some titles may have missing `genre_tmdb`; genre filtering should not drop those titles unless explicitly filtered.
- Provider lookup may be missing for some ids; popularity mart should keep counts even if provider names are NULL.
- Dataset period is not a full year; marts must avoid ‚Äúyearly‚Äù naming/claims and expose dataset min/max timestamps.

## Out of Scope

- WhatsApp/Telegram bot integration, message handling, and recommendation ranking logic beyond telemetry-derived popularity.
- Personalization (user embeddings, collaborative filtering) and any Cortex/LLM enrichment steps.

## Open Questions

- Which source event table should be the default for runs (T1 vs T4) depends on cost/latency needs; this spec defaults to T1 via `vars.events_model`, with explicit override for larger scopes.
